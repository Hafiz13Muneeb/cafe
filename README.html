Overview

CafeFlow is a turnkey digital menu solution designed for independent cafés and restaurants. It replaces physical menus with a QR‑code‑powered web experience — customers scan, browse, and place orders directly via WhatsApp. The owner manages everything through a secure admin dashboard.

This repository contains the full‑stack implementation for a single‑cafe deployment. It eliminates complex multi‑tenancy, subscription fees, and third‑party ordering commissions. The system auto‑seeds a default cafe on first boot, so you can be live in minutes.

⚡ Key differentiator — orders are sent directly to the owner’s WhatsApp. No middleware, no per‑order fees, zero commission.

Features

📱 Digital Menu

Real‑time menu updates, category filtering, and availability toggles. Instantly reflects changes.

🧾 WhatsApp Ordering

Customers select a table, review their cart, and send a formatted order directly to the cafe WhatsApp.

📊 Built‑in Analytics

Track views, order attempts, completed orders, and revenue. Busiest days & top items insights included.

🎨 Full Theming

Customise primary/secondary colours, light/dark mode, logo, and favicon — all from the admin panel.

🔐 Secure Admin

Environment‑based authentication with password change & session persistence via localStorage.

🖨️ QR Code Generator

Generate and download high‑resolution QR codes for your menu. Place them on every table.

📋 FAQ Manager

Create, edit, reorder, and enable/disable customer FAQs. Shown in the public help widget.

📦 Auto‑Seed

On first start, the database is seeded with a default cafe owner and sample menu items. Zero manual setup.

Architecture Overview

The system follows a decoupled client‑server pattern. The frontend is a single‑page application served statically; the backend provides a RESTful API. Both communicate via HTTP, with credentials (cookies) used for session affinity.

  • Authentication: Compares credentials against .env. No JWT is issued; the backend relies on the protect middleware which attaches the first (and only) cafe owner from the database to req.user.
  • Public Menu Resolution: The GET /api/menu/:slug endpoint uses a smart fallback: try exact slug → first cafe with items → first cafe in DB.
  • Analytics Pipeline: Events (view, order_attempt, order_completed) are stored in a lightweight MongoDB collection. Aggregations are performed on‑the‑fly for the dashboard charts.
  • Image Lifecycle: Uploaded images are stored on Cloudinary. On update or deletion, the old image is automatically purged from the CDN.

Getting Started

Prerequisites

  • Node.js ≥ 18.x
  • MongoDB (Atlas or local)
  • Cloudinary account (for image uploads)
  • Package manager: npm or yarn

1. Clone the Repository

git clone https://github.com/your-org/cafe-flow.git
cd cafe-flow

2. Backend Setup

cd server
cp .env.example .env        # or create .env manually
npm install

Edit .env with your MongoDB URI, Cloudinary credentials, and owner credentials. See the Environment Variables section for details. 

npm run dev     # starts backend on http://localhost:5000

3. Frontend Setup

cd ../client
cp .env.example .env
npm install
npm run dev     # starts Vite dev server on http://localhost:5173

4. First Boot & Seeding

The backend automatically seeds a default cafe owner and sample menu items if the database is empty. You’ll see this in the console: 

🔄 Seeding default cafe owner...
✅ Cafe owner created: admin (My Cafe)
✅ 8 sample menu items added.
🔑 Login: admin / admin123
🔗 Public Menu: /menu/cafe

You can change the default credentials via the .env file before the first start.

Deployment

Production Build

# Build frontend
cd client
npm run build        # outputs to client/dist

# Backend serves the built frontend in production
cd ../server
NODE_ENV=production npm start

The backend is configured to serve static files from client/dist when NODE_ENV=production. All routes except /api/* fall back to index.html.

Hosting Recommendations

  • Backend: Railway, Render, Heroku, or any Node.js hosting.
  • Database: MongoDB Atlas (preferred for production).
  • Media: Cloudinary (already integrated).
  • Frontend: Can be served by the backend itself, or separately on Netlify/Vercel (ensure API base URL is configured).

Pro‑tip: For production, set NODE_ENV=production, enable MongoDB Atlas IP whitelist, and ensure Cloudinary environment variables are properly set.

Troubleshooting

❌ MongoDB connection string is missing
Ensure MONGO_URI, MONGODB_URI, or MONGO_URL is set in your .env.
❌ Public menu returns 404 even with DB seeded
Check the slug in the URL. Default is /menu/cafe. Verify the owner has a slug field.
❌ Cloudinary upload fails with 401
Double‑check CLOUDINARY_CLOUD_NAME, API_KEY, and API_SECRET. Ensure the cloud name is correct.
❌ Login fails even with correct credentials
Make sure the frontend VITE_OWNER_USERNAME and VITE_OWNER_PASSWORD exactly match the backend variables. If you changed them after seeding, update both.
❌ Cart doesn't sync across tabs
The app uses localStorage with storage events. Ensure both tabs are on the same origin and no browser extensions block storage events.
