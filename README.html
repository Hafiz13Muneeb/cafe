📱 QR Menu SaaS

v2.0 – Multi‑Tenant Lite

A closed B2B SaaS platform for cafés and restaurants. Each café gets its own isolated menu, theme, and QR code – all powered by a super‑admin control panel. Customers order via WhatsApp with zero commission.

✨ Key Features

  • Multi‑Tenancy – every café owner has isolated data (menu items, settings, orders).
  • Super‑Admin Dashboard – create, block, edit, and delete café owners.
  • Dynamic Customer Routing – each café gets a unique slug (/menu/:slug).
  • QR Code Generation – QR codes encode the café’s specific slug URL.
  • Database‑Driven Theming – owners choose primary/secondary colors and dark/light mode; CSS variables update instantly.
  • WhatsApp Ordering – customers select a table and the order is sent directly to the café’s WhatsApp.
  • Zero Commission – no per‑transaction fees; pure WhatsApp routing.

🧩 Architecture Overview

The system is built around a single User model that supports two roles: superadmin and owner. Owners have café‑specific fields such as cafeName, slug, whatsappNumber, tables, and a theme object. All menu items are linked to an ownerId to ensure strict data isolation.

The frontend uses React Context for auth, cart, and theme state. The ThemeContext applies CSS variables to the root element, so the customer menu dynamically adapts to each café’s theme.

🔐 Authentication & Roles

  • Login – /api/auth/login (username + password).
  • JWT – stored in localStorage as adminToken.
  • Middleware – protect verifies token and attaches user; restrictTo limits access by role.
  • Blocking – if isBlocked: true, the protect middleware returns 403 and the customer menu shows a “cafe unavailable” message.

🗄️ Database Models (Mongoose)

User (covers both superadmin and owners)

{
  username: String, unique
  email: String, unique
  password: String (hashed)
  role: 'superadmin' | 'owner'
  isBlocked: Boolean
  // Owner-specific
  cafeName: String
  slug: String (unique, used for public URL)
  whatsappNumber: String
  logoUrl: String
  faviconUrl: String
  tables: [String]
  theme: {
    primaryColor: String (hex)
    secondaryColor: String (hex)
    mode: 'light' | 'dark'
  }
}

MenuItem

{
  ownerId: ObjectId (ref: User)
  title: String
  description: String
  price: Number
  category: String
  imageUrl: String (Cloudinary URL)
  isAvailable: Boolean
}

🚀 Installation & Setup

Prerequisites

  • Node.js ≥ 18
  • MongoDB (local or Atlas)
  • Cloudinary account (for image uploads)

1. Clone & Install

git clone <your-repo>
cd cafe-menu
npm install --prefix backend
npm install --prefix frontend

2. Environment Variables

Create backend/.env:

PORT=5000
MONGO_URI=mongodb://localhost:27017/cafe-saas
JWT_SECRET=your-super-secret-key
CLOUDINARY_CLOUD_NAME=your-cloud-name
CLOUDINARY_API_KEY=your-api-key
CLOUDINARY_API_SECRET=your-api-secret
CLIENT_URL=http://localhost:5173

3. Seed Superadmin

On first start, the server auto‑creates a superadmin with:

Username: superadmin
Password: SuperAdmin@2026!

⚠️ Change this password immediately after first login.

4. Run Development

# Backend (port 5000)
cd backend
npm run dev

# Frontend (port 5173)
cd frontend
npm run dev

Access the app at http://localhost:5173.

🎨 Dynamic Theming

The customer menu fetches the café’s theme from the backend and injects CSS variables: 

--primary-color: #d4a843;
--secondary-color: #b8860b;
--bg-color: #f8fafc;
--text-color: #0f172a;
--card-bg: #ffffff;
--border-color: #e2e8f0;

All components use these variables, so changing the database values instantly updates the entire customer experience.

🔧 Customization & Extending

  • Add new theme properties – extend the theme object in User schema and update ThemeContext.
  • Add additional roles – modify the role enum and update the restrictTo middleware.
  • Integrate payment gateway – extend the cart/order flow to support online payments.
  • Multi‑language support – use i18n libraries and store language preference in settings.

📦 Deployment

Build the frontend and serve it via the backend: 

cd frontend
npm run build
# The backend server.js will serve the built files in production

Set NODE_ENV=production and deploy the backend to your hosting provider.
